# Architecture Decision Records

ADR — фиксация архитектурного решения в тот момент, когда оно
принято. Не справочник и не гайд: справочник расскажет КАК, ADR —
ПОЧЕМУ именно так, а не иначе, и что мы при этом отвергли.

Перед тем как переспрашивать "почему у нас X, а не Y" — читаем
соответствующий ADR.

## Когда заводить новый ADR

- Вводим новое значение в `AgentPhase` (новая фаза хода).
- Меняем направление sync (например, round-trip workspace↔агенты).
- Отказываемся от SemVer для схем.
- Добавляем event bus / прямые вызовы агент↔агент.
- Меняем формат персистентности (JSON → SQLite и т.п.).
- Любая правка, которая инвалидирует другой ADR (тогда пишем новый со
  ссылкой `Supersedes: NNNN`).

Если сомневаешься — лучше завести.

## Правила

1. Нумерация сквозная, zero-padded: `0001`, `0002`, …, `0042`.
2. Имя файла — `NNNN-kebab-case-title.md`.
3. Шаблон — `TEMPLATE.md` рядом.
4. Статусы: `proposed` / `accepted` / `superseded by NNNN` / `deprecated`.
5. Принятый ADR не редактируем по сути. Меняется решение — пишется новый.
6. Мелкие правки (typo, ссылки) — можно в существующем.
