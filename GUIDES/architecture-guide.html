worldsim — архитектура MVP

Текстовая RPG с миром, который генерируется LLM по ходу игры. Канон хранится на диске, между сессиями не теряется. MVP-контур: ролевая игра + исследование (без боёвки).

Цель

  • Игрок играет в терминале на русском, общается свободным текстом.
  • Мир живёт: NPC помнят разговоры, фракции реагируют на репутацию, секреты постепенно раскрываются.
  • Hard-ограничения (не телепортируешься, не используешь предметы, которых нет) проверяются детерминированно.
  • Soft-ограничения (тон, драматургия, настроение NPC) ведёт LLM.

Цикл хода (13 шагов)

  1. build_turn_context(snapshot) — вырезаем из канона только то, что нужно (текущая локация, соседи, 5 ближайших NPC, активные арки, недавние события timeline).
  2. parse_intent(player_input) — LLM нормализует свободный текст в Intent.
  3. validate_intent — Python-проверки: target существует, предмет в инвентаре, локации связаны, condition позволяет.
  4. Если intent = converse и есть целевой NPC → npc-mind.run.
  5. world-builder.run_turn_update — патч мира, narrative_summary.
  6. personal-progression.run — патч игрока.
  7. canon-keeper.validate(mode="patch"). Если не ok — блокируем, откатываемся.
  8. apply_turn — сохраняем патчи + new_facts + timeline_event.
  9. save_world — дамп на диск.
  10. scene-master.run — рендер игроку.
  11. append_session_log.
  12. Вывод в терминал.
  13. Ждём следующего ввода.

Канон — четыре слоя правды

  • Онтологический — что есть в мире (locations, characters, factions, secrets).
  • Эпистемический — кто что знает (Character.knowledge, Secret.known_by, PlayerProgression.known_facts).
  • Нарративный — что произошло (timeline.jsonl, session_log.jsonl).
  • Игрок-facing — что игрок видит (рендер scene-master с эпистемическим фильтром).

Зоны ответственности (кто что пишет)

Главный принцип: у каждого файла есть ровно один писатель. Никаких гонок.

  • locations / characters[NPC] / factions / secrets / arcs / plot_state: world-builder.
  • player_progression: personal-progression.
  • characters[player] (основные поля): world-builder при init; потом не меняется (атрибуты/скиллы живут в player_progression).
  • timeline: orchestrator аппендит (из TurnPatch.timeline_event).
  • settings: orchestrator (через CLI).
  • session_log: orchestrator.

Hard vs soft constraints

Hard (Python, deterministic, всегда):

  • target должен существовать в канон
  • предмет для действия — в inventory
  • локация-цель — связана с текущей (Location.connections)
  • NPC, к которому обращаются, — в той же или соседней локации
  • знание факта — проверяется по Character.knowledge / player_progression.known_facts
  • condition dead / exhausted блокирует risk_level high

Soft (LLM, pydantic-клампы на диапазоны):

  • attitude_to_player ∈ [−1, 1]
  • dramatic_pressure ∈ [0, 1]
  • attributes.* ∈ [0, 1]
  • reputation[faction] ∈ [−1, 1]
  • Arc.progress ∈ [0, 1], монотонно вверх (кроме reversal)
  • Secret.status ∈ {hidden, hinted, revealed} — только вверх

Монотонные инварианты

  • Arc.progress, Arc.clarity_to_player — только вверх (кроме stage=reversal).
  • Secret.status: hidden → hinted → revealed, без откатов.
  • known_facts: только добавление; забывание — через явное событие (в MVP запрещено).
  • condition: dead — финально. Воскрешения запрещены.
Упражнение: прочитайте docs/constraints.md, потом код worldsim-canon-keeper/src/worldsim_canon_keeper/agent.py — увидите все эти правила как Python.

Ленивая генерация

При world_init мы создаём только минимум: 1 поселение, 4–6 локаций, 2–3 фракции, 5–8 NPC, 2–3 секрета, 1–2 арки. full_description у локаций — null.

Когда игрок впервые заходит в локацию с full_description=null, orchestrator вызывает world-builder.run_location_detail, сохраняет результат в канон. Это — единственное место, где канон расширяется между ходами.

Upgrade path (после MVP)

  • Агенты — in-process pip-пакеты. Если понадобится — можно обернуть в FastAPI/MCP и поднять в контейнерах.
  • Канон — JSON-файлы. Масштабируется в SQLite/Postgres без изменения схем (они pydantic, сериализация — отдельная фаза).
  • Бой, долгие квесты, мультиплеер — не ломают ядро, достраиваются новыми агентами.
