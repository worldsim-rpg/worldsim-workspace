worldsim — руководство разработчика

Два режима: Новичок — только то, что нужно знать сразу. Разработчик — те же разделы с полными деталями, инструментами и edge-case. Усвоили базу — переключитесь и прочитайте, что стоит за каждым пунктом.

0. Как это устроено

Игра worldsim — 7 независимых git-репо. Один из них — worldsim-workspace — ваша рабочая папка с документацией и общими пакетами. Остальные шесть — отдельные агенты (мозги игры): оркестратор, мир-движок, хранитель канона, рассказчик, диалог NPC, прогрессия игрока.

Агенты общаются не по сети, а как pip-пакеты внутри одного процесса. Никаких контейнеров и очередей — просто функции.

Архитектура целиком → architecture-guide.html. Прочитайте её первой.

Игра — 7 независимых git-репо. worldsim-workspace — source of truth: pydantic-схемы канона, базовый prompt-клиент, sync-скрипты, HTML-гайды. Шесть агент-репо — stateless callable'ы, установленные в editable-режиме.

Поток хода: orchestrator принимает ввод игрока → валидирует intent (hard + soft constraints) → последовательно вызывает нужные агенты → агенты читают/пишут canonical state через структуры из worldsim_schemas → orchestrator сохраняет результат и возвращает сцену.

Агенты stateless: они не хранят ничего между ходами. Весь state — в save-файле, который orchestrator передаёт в каждый вызов.

Детально: architecture-guide.html, docs/loop.md (13 шагов хода), docs/agent-map.md (кто что пишет в канон).
Хотите больше? В режиме Разработчик — поток хода пошагово, stateless-модель и почему это важно.

1. Первый запуск

1.1. Клонируйте 7 репо рядом

mkdir worldsim && cd worldsim
for r in worldsim-workspace worldsim-orchestrator worldsim-world-builder \
         worldsim-canon-keeper worldsim-scene-master worldsim-npc-mind \
         worldsim-personal-progression; do
  git clone "https://github.com/worldsim-rpg/$r.git"
  git -C "$r" checkout dev
done

Все семь — на одном уровне. Sync-скрипт ищет их как соседние папки.

1.2. Python 3.11+, venv

cd worldsim-workspace
python -m venv .venv
.venv\Scripts\activate      # Windows
source .venv/bin/activate   # macOS/Linux

1.3. Установка

pip install -e packages/schemas
pip install -e packages/prompts
pip install -e ../worldsim-world-builder
pip install -e ../worldsim-canon-keeper
pip install -e ../worldsim-scene-master
pip install -e ../worldsim-npc-mind
pip install -e ../worldsim-personal-progression
pip install -e ../worldsim-orchestrator   # последним

1.4. API-ключ

cp ../worldsim-orchestrator/.env.example ../worldsim-orchestrator/.env
# откройте .env и впишите ANTHROPIC_API_KEY

1.5. Запуск

worldsim new                 # создать мир
worldsim play --world <id>   # играть
worldsim worlds              # список миров

1.1. Клон + ветка dev

mkdir worldsim && cd worldsim
for r in worldsim-workspace worldsim-orchestrator worldsim-world-builder \
         worldsim-canon-keeper worldsim-scene-master worldsim-npc-mind \
         worldsim-personal-progression; do
  git clone "https://github.com/worldsim-rpg/$r.git"
  git -C "$r" checkout dev
done

Все семь — на одном уровне. Sync ищет соседей по ../worldsim-*.

Ветки: dev (прямой push разработчику) → staging (PR с апрувом) → main (деплой). Push в staging/main заблокирован branch protection.

1.2. Editable install

cd worldsim-workspace
python -m venv .venv && source .venv/bin/activate   # или Scripts\activate на Windows
pip install -e packages/schemas
pip install -e packages/prompts
pip install -e ../worldsim-world-builder
pip install -e ../worldsim-canon-keeper
pip install -e ../worldsim-scene-master
pip install -e ../worldsim-npc-mind
pip install -e ../worldsim-personal-progression
pip install -e ../worldsim-orchestrator

-e (editable) означает: изменения в src/ подхватываются без переустановки. Orchestrator ставится последним — он тянет typer, rich, anthropic и регистрирует CLI-точку входа.

1.3. Переменные окружения

cp ../worldsim-orchestrator/.env.example ../worldsim-orchestrator/.env
# .env читается через python-dotenv при старте orchestrator

Для отладки: WORLDSIM_DEBUG=1 worldsim play --world <id> — в stderr будет raw-ответ LLM перед JSON-парсингом.

1.4. Проверка установки

python -c "import worldsim_schemas; print(worldsim_schemas.__version__)"
python -m pytest -q packages/schemas/   # убедиться, что схемы чистые
Хотите больше? В режиме Разработчик — зачем -e, как работает WORLDSIM_DEBUG, и как проверить установку без запуска игры.

3. Где правят — железное правило

90% изменений — это правка markdown-промптов, не Python. Если Claude Code хочет залезть в .py — сначала спросите: точно ли это не промпт?

Каждый агент: код в src/.../agent.py — тонкий клей «возьми input → подставь в промпт → спарси JSON». Поведение живёт в prompts/*.md.

90% изменений — промпты, не Python.

Структура каждого агент-репо:

worldsim-<agent>/
├── src/worldsim_<agent>/
│   └── agent.py        ← тонкий клей: IO + prompt call + JSON parse
├── prompts/
│   ├── system.md       ← system prompt (роль агента)
│   └── *.md            ← операционные промпты (turn, init, …)
├── _schemas/           ← снимок схем (не редактировать руками)
└── _prompts/           ← снимок prompt-утилит (не редактировать руками)

Контракт agent.py: принимает pydantic-объект из worldsim_schemas, формирует строку промпта, вызывает anthropic.messages.create(), парсит JSON-ответ обратно в pydantic-объект и возвращает его orchestrator'у. Никакой бизнес-логики в agent.py — только клей.

Если нужно изменить что агент делает — правьте промпт. Если нужно изменить как передаётся данные между агентами — правьте схему + agent.py + промпт.

Хотите больше? В режиме Разработчик — полная структура агент-репо и контракт agent.py.

4. Как добавить фичу, не сломав всё

  1. Найдите зону. Откройте docs/agent-map.md — там таблица, кто что пишет в канон. Фича должна полностью лежать в одной зоне.
  2. Нужно новое поле? Правьте схему в workspace → sync → поддержка в агентах → промпт.
  3. Новое ограничение? Добавьте функцию в validators.py + тест.
  4. Новое поведение NPC / soft-характеристика? Только промпты, никакого кода.

4.1. Определите зону

Откройте docs/agent-map.md. Колонки: агент → какие поля канона он пишет. Фича должна полностью укладываться в одну зону. Если тянет в несколько — разделите на атомарные изменения и заводите отдельные PR.

4.2. Изменение схемы (новое поле в сущности)

  1. Правка в packages/schemas/src/worldsim_schemas/schemas.py.
  2. python -m pytest -q packages/schemas/.
  3. ./repo-setup/sync-all.sh — раскатит новые схемы в _schemas/ каждого агент-репо.
  4. В каждом агент-репо, где agent.py читает/пишет это поле — добавьте поддержку.
  5. Промпт, который должен генерировать это поле — правка prompts/*.md.
Никогда не правьте _schemas/ внутри агент-репо руками. Это снимок. Единственный путь — workspace → sync.

4.3. Новая hard-constraint

Живут в worldsim-orchestrator/src/.../validators.py. Каждая — маленькая функция, возвращает ConstraintResult(ok, code, message). Добавьте вызов в validate_intent.

Тесты обязательны: tests/test_validators.py. Один позитивный + один негативный на каждый constraint.

4.4. Новое поведение NPC / soft-характеристика

Только промпты:

  1. Найдите промпт, который генерирует это поведение (обычно turn_update.md или world_init.md).
  2. Добавьте в раздел "Rules" новый пункт.
  3. Если нужно новое поле — сначала п. 4.2.
  4. Запустите worldsim new, создайте свежий мир, проверьте живьём. Не тестируйте на старых сейвах.
Хотите больше? В режиме Разработчик — пошагово: схема → sync → agent.py → промпт, как писать constraints с тестами, почему нельзя тестировать на старых сейвах.

5. Что запрещено

  • Один файл канона — один писатель. Если world-builder хочет записать что-то в player_progression — это ошибка.
  • Не добавляйте новых агентов без веской причины. Сейчас их 6.
  • Не ломайте монотонность инвариантов: progress, known_facts, Secret.status, condition=dead — только в одну сторону.
  • Не меняйте схему сейва без бэксовместимости. Старые сохранения не должны умирать после git pull.
  • Один файл канона — один писатель. Почему: если два агента пишут в одно поле, orchestrator не может детерминировано разрешить конфликт. Всегда добавляйте патч в зону ответственного агента.
  • Не добавляйте новых агентов без веской причины. Почему: каждый новый агент — новый LLM-вызов, новый источник расхождений, новое место для JSON-ошибки. Если фича укладывается в существующий агент — кладите туда.
  • Монотонные инварианты нельзя обращать вспять. Поля progress, known_facts, Secret.status, condition=dead — только в одну сторону. Почему: consistency-проверки canon-keeper строятся на этом допущении; нарушение создаёт трудноуловимые расхождения.
  • Схема сейва — только с бэксовместимостью. Новые поля — через Optional[T] = None или default-значения в pydantic. Почему: у пользователя могут быть сейвы, которые он не хочет потерять после git pull. Смотрите docs/schema-versioning.md за деталями миграции.
Хотите больше? В режиме Разработчик — почему каждый запрет существует и как правильно обходить ограничения при необходимости.

6. Поток работы через Claude Code

  1. Опишите, что хотите. Не «исправь файл X», а «хочу, чтобы NPC запоминали ложь игрока».
  2. Попросите сначала предложить план. Не сразу писать код.
  3. План должен назвать: какой агент, какие промпты, нужна ли схема, какие тесты.
  4. Если план затрагивает больше одного агента — насторожитесь. Большинство правок одноагентные.
  5. После правки: sync-all.sh (если меняли схемы) + pytest -q.
  6. Запустите worldsim new, проверьте на свежем мире.
  1. Формулируйте через поведение, не через файлы. «NPC запоминают ложь и снижают attitude» → Claude сам найдёт, где это реализовать.
  2. Plan first. Явно просите: «предложи план, не пиши код». Хороший план: агент → промпты → схема (если нужна) → тесты → порядок шагов.
  3. Один агент за раз. Если план касается двух агентов — попросите разбить на два отдельных PR. Это упрощает ревью и откат.
  4. Issue перед правкой. Создайте issue в нужном репо: gh issue create --repo worldsim-rpg/worldsim-<agent> --title "...". Это ваш чекпоинт и история решений.
  5. После правки: ./repo-setup/sync-all.sh (если менялись схемы) + python -m pytest -q во всех задетых репо.
  6. Тест на свежем мире. worldsim new, не старый сейв. Старые сейвы могут не иметь новых полей — тест будет нечестным.
  7. PR-поток: коммит в dev → PR dev → staging с описанием изменений → апрув → мерж → PR staging → main для деплоя.
Хотите больше? В режиме Разработчик — как создавать issue через CLI, PR-поток dev→staging→main, почему нельзя тестировать на старых сейвах.

7. Как обновляться

cd worldsim/
for d in worldsim-*; do
  (cd "$d" && git pull --ff-only)
done

cd worldsim-workspace
./repo-setup/sync-all.sh

После этого перезапустите тесты. Если что-то упало — смотрите последний git log в workspace.

cd worldsim/
for d in worldsim-*; do
  (cd "$d" && git fetch && git pull --ff-only origin dev)
done

cd worldsim-workspace
./repo-setup/sync-all.sh
python -m pytest -q packages/schemas/

# тесты во всех агентах
for d in ../worldsim-{orchestrator,world-builder,canon-keeper,scene-master,npc-mind,personal-progression}; do
  echo "=== $d ===" && python -m pytest -q "$d"
done

Если упал тест после обновления схем: скорее всего в агенте есть agent.py который не поддерживает новое обязательное поле. Смотрите последний коммит в workspace: git -C worldsim-workspace log --oneline -5.

Совместимость сейвов: если схема изменилась не по-backward-compatible, старые сейвы не загрузятся. Проверьте docs/schema-versioning.md — там описан процесс миграции через migrate_save().

Хотите больше? В режиме Разработчик — как запустить тесты во всех агентах разом, что делать с несовместимыми сейвами после обновления схем.
