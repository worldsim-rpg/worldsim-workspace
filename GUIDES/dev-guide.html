Вы делегируете Клоду всё, что можно. Ваша задача — понимать, что просить, и проверять, что получилось. Код руками писать не нужно.

0. За одну минуту: что это такое

Игра состоит из семи независимых частей — агентов. Каждый агент отвечает за свою зону: один генерирует мир, другой следит за историей, третий ведёт диалоги NPC и так далее. Они работают вместе, не видя внутрь друг друга.

Поведение агентов задаётся текстовыми инструкциями — промптами. Это обычные файлы на обычном языке. Когда вы хотите что-то изменить в игре, почти всегда достаточно поправить один такой файл — и Клод сделает это за вас.

Хотите понять архитектуру глубже — architecture-guide.html.

1. Первый запуск

Перед началом у вас должны быть:

  • Python 3.11 или новее
  • API-ключ Anthropic (получить на console.anthropic.com)
  • Аккаунт на GitHub

Скажите Клоду:

Склонируй все семь worldsim-репо из организации worldsim-rpg на GitHub, настрой виртуальное окружение Python, установи все зависимости и создай файл с API-ключом. Ключ: вставьте сюда.

Когда Клод закончит, проверьте что всё работает:

Запусти worldsim new и создай тестовый мир.

Если появились ошибки — скопируйте текст ошибки и отдайте Клоду.

2. Когда что-то не работает

Почти всегда достаточно описать симптом Клоду — он разберётся.

Что происходитЧто сказать Клоду
Игра не запускается, ошибка при старте «Скопируй ошибку и разберись с установкой»
Игра говорит «не могу» на каждое действие «Найди, какое ограничение срабатывает, и объясни мне»
NPC ведёт себя не так, как хотелось бы «Найди промпт, который управляет поведением NPC в диалоге»
Сцены скучные или слишком длинные «Найди промпт рассказчика и покажи мне, что там написано»
Что-то сломалось после изменений «Откати последнее изменение в [название репо]»

3. Где живут изменения

Запомните одно правило: поведение игры — это текст, не код. Почти всё, что вы хотите изменить, лежит в файлах prompts/*.md внутри каждого агента. Это обычный текст с инструкциями.

Код трогать не нужно. Когда Клод предлагает изменить .py-файл — уточните, точно ли это не промпт:

Ты уверен, что это нужно менять в коде, а не в промпте?

4. Как попросить добавить фичу

Рабочий порядок — сначала план, потом правки:

  1. Опишите желаемое поведение. Не «исправь файл X» — а что должно измениться в игре:
    Хочу, чтобы NPC запоминали, когда игрок им врёт, и в следующий раз встречали его с недоверием.
  2. Попросите план.
    Предложи план изменений. Пока ничего не меняй — только объясни, что и где нужно поправить.
  3. Проверьте план. Хороший знак — изменения касаются одного агента и одного-двух промпт-файлов. Плохой знак — план затрагивает много репо сразу. Тогда скажите:
    Разбей на отдельные шаги, по одному агенту за раз.
  4. После правок — проверьте живьём.
    Создай новый тестовый мир и проверь, что изменение работает.

5. Что нельзя просить Клода делать

Есть несколько запретов, которые нужно держать в голове. Если Клод предлагает что-то из списка — остановите его.

  • Не давайте двум агентам писать в одно и то же место. Если план говорит «world-builder запишет это в player_progression» — это ошибка. Скажите Клоду: «Это должен делать только тот агент, который отвечает за эту зону».
  • Не добавляйте новых агентов. Их шесть — этого достаточно. Любую фичу можно вписать в существующего.
  • Не ломайте старые сохранения. Если план меняет структуру данных игры — спросите Клода: «Старые сейвы будут работать после этого изменения?»

6. Рабочий ритм

  1. Описываете желаемое поведение → просите план → апрувите план.
  2. Клод делает правки → запускает тесты → создаёт новый тестовый мир.
  3. Вы проверяете игру. Если не то — описываете разницу.
  4. Клод коммитит и отправляет изменения в ветку dev.
Полезная фраза для любой задачи:
«Предложи план, объясни какие файлы затронешь и почему, потом жди моего подтверждения перед тем как начать».

0. Архитектура за две минуты

Семь git-репо. worldsim-workspace — source of truth: pydantic-схемы канона, базовый prompt-клиент, sync-скрипты. Шесть агент-репо — stateless callable'ы, установленные через pip install -e.

Поток хода: orchestrator принимает ввод → валидирует intent (hard + soft constraints) → вызывает нужные агенты → агенты читают/пишут worldsim_schemas-объекты → orchestrator сохраняет state и возвращает сцену.

Агенты stateless — весь state в save-файле, который orchestrator передаёт на каждый вызов.

architecture-guide.html — полная архитектура.
docs/loop.md — 13 шагов хода.
docs/agent-map.md — кто что пишет в канон.

1. Первый запуск

Клон + ветки

mkdir worldsim && cd worldsim
for r in worldsim-workspace worldsim-orchestrator worldsim-world-builder \
         worldsim-canon-keeper worldsim-scene-master worldsim-npc-mind \
         worldsim-personal-progression; do
  git clone "https://github.com/worldsim-rpg/$r.git"
  git -C "$r" checkout dev
done

Все семь — на одном уровне. Sync ищет соседей по ../worldsim-*.

Ветки: dev (прямой push) → staging (PR + апрув) → main (деплой). Push в staging/main заблокирован branch protection.

Editable install

cd worldsim-workspace
python -m venv .venv && source .venv/bin/activate   # Windows: Scripts\activate
pip install -e packages/schemas
pip install -e packages/prompts
pip install -e ../worldsim-world-builder
pip install -e ../worldsim-canon-keeper
pip install -e ../worldsim-scene-master
pip install -e ../worldsim-npc-mind
pip install -e ../worldsim-personal-progression
pip install -e ../worldsim-orchestrator   # последним — регистрирует CLI

ENV + проверка

cp ../worldsim-orchestrator/.env.example ../worldsim-orchestrator/.env
# вписать ANTHROPIC_API_KEY

python -c "import worldsim_schemas; print(worldsim_schemas.__version__)"
python -m pytest -q packages/schemas/

Отладка: WORLDSIM_DEBUG=1 worldsim play --world <id> — в stderr raw-ответ LLM перед JSON-парсингом.

2. Когда что-то не работает

СимптомПричинаДиагностика
ImportError при старте Пакет не установлен или другой venv pip list | grep worldsim — должны быть все 8.
"ANTHROPIC_API_KEY not set" .env не найден Проверьте worldsim-orchestrator/.env, venv активен?
Hard-validator блокирует validators.py вернул ok=False WORLDSIM_DEBUG=1 — покажет code и message из ConstraintResult.
NPC знает лишнее Широкий knowledge при init или утечка в промпт Проверьте world_init.md — какие факты попадают в knowledge.
Сцены сухие / длинные Инструкции по тону в render.md Правьте раздел "Style rules", запускайте worldsim new.
Бесконечный грайнд скилла Пороги в progression.md Раздел "Thresholds" — XP-значения и decay-коэффициент.
JSON-парс упал LLM обернул ответ в markdown-блок WORLDSIM_DEBUG=1 → raw-ответ. Добавьте "Return only valid JSON, no markdown fences" в конец промпта.

Принцип: сначала WORLDSIM_DEBUG=1, потом промпты, потом agent.py.

3. Структура агента и где правят

worldsim-<agent>/
├── src/worldsim_<agent>/
│   └── agent.py        ← IO + prompt call + JSON parse, бизнес-логики нет
├── prompts/
│   ├── system.md       ← system prompt (роль агента)
│   └── *.md            ← операционные промпты (turn, init, …)
├── _schemas/           ← снимок схем (sync-only, руками не трогать)
└── _prompts/           ← снимок prompt-утилит (sync-only)

Контракт agent.py: принимает pydantic-объект из worldsim_schemas → формирует промпт → вызывает anthropic.messages.create() → парсит JSON обратно в pydantic → возвращает orchestrator'у. Никакой бизнес-логики.

Менять поведение → промпт.
Менять передаваемые данные → схема + agent.py + промпт.
Лезть в agent.py без изменения схемы — почти всегда ошибка.

4. Как добавить фичу

4.1. Определите зону

docs/agent-map.md — таблица: агент → какие поля канона пишет. Фича должна полностью лежать в одной зоне. Если тянет в несколько — разбейте на атомарные изменения, отдельные PR.

4.2. Новое поле в схеме

  1. Правка в packages/schemas/src/worldsim_schemas/schemas.py.
  2. python -m pytest -q packages/schemas/.
  3. ./repo-setup/sync-all.sh — раскатит в _schemas/ всех агентов.
  4. В каждом agent.py, который читает/пишет это поле — добавить поддержку.
  5. Промпт, который генерирует поле — правка prompts/*.md.
Никогда не правьте _schemas/ руками. Только workspace → sync.

4.3. Новая hard-constraint

worldsim-orchestrator/src/.../validators.py — каждая constraint возвращает ConstraintResult(ok, code, message), вызов добавляется в validate_intent.

Тесты обязательны: tests/test_validators.py. Один позитивный + один негативный на каждый constraint.

4.4. Новое поведение / soft-характеристика

Только промпты. Найдите промпт (обычно turn_update.md или world_init.md), добавьте пункт в раздел "Rules". Если нужно новое поле — сначала п. 4.2. Тестируйте на свежем worldsim new, не на старых сейвах.

5. Что запрещено и почему

  • Один файл канона — один писатель. Два агента в одном поле → orchestrator не может детерминировано разрешить конфликт.
  • Не добавляйте новых агентов. Каждый новый — новый LLM-вызов, новый источник расхождений, новое место для JSON-ошибки.
  • Монотонные инварианты нельзя обращать. progress, known_facts, Secret.status, condition=dead — только в одну сторону. Canon-keeper строит consistency-проверки на этом допущении.
  • Схема сейва — только с бэксовместимостью. Новые поля через Optional[T] = None или pydantic-default. Подробности: docs/schema-versioning.md.

6. Рабочий ритм

  1. Issue перед правкой: gh issue create --repo worldsim-rpg/worldsim-<agent> --title "..."
  2. Plan first. Агент → промпты → схема (если нужна) → тесты → порядок шагов. Один агент за раз.
  3. После правок: ./repo-setup/sync-all.sh (если менялись схемы) + python -m pytest -q во всех задетых репо.
  4. Тест на свежем мире: worldsim new.
  5. PR-поток: коммит в dev → PR dev → staging (описание + апрув) → PR staging → main для деплоя.

7. Как обновиться

cd worldsim/
for d in worldsim-*; do
  (cd "$d" && git fetch && git pull --ff-only origin dev)
done

cd worldsim-workspace
./repo-setup/sync-all.sh

for d in . ../worldsim-{orchestrator,world-builder,canon-keeper,scene-master,npc-mind,personal-progression}; do
  echo "=== $d ===" && python -m pytest -q "$d"
done

Тест упал после обновления схем → скорее всего агент не поддерживает новое обязательное поле. Смотрите: git -C worldsim-workspace log --oneline -5.

Несовместимые сейвы → docs/schema-versioning.md — процесс миграции через migrate_save().
